1. Présentation du Projet
1.1 Objectif général
Développer une application web SaaS pour centraliser la gestion d'une exploitation avicole couplée à un
réseau de boutiques. La solution couvre la production, la santé animale, l'alimentation, les finances, le
commerce, les charges et le reporting.

2. Principe Fondateur — Table mouvements_caisse
Universelle
2.1 Concept
TOUS les flux d'argent transitent par une seule table : mouvements_caisse. Caisses physiques et comptes
bancaires sont reunis dans une seule table portefeuilles. source_id et dest_id pointent tous les deux vers
portefeuilles.id — le champ type (caisse ou banque) distingue les deux.
Chaque mouvement est un transfert entre une SOURCE et une DESTINATION :

un tiers exterieur (fournisseur, client, veterinaire...) — source_type / dest_type = 'tiers'

4. Module COFO
4.1 Definition
Le COFO est un produit naturel issu des arbres, utilise pour proteger les poules contre la fraicheur. Il peut
etre collecte, mis en sac et revendu. Il n'est pas gere en stock interne : seuls les flux financiers sont suivis.

6. Module Alimentation
6.1 Principe
Ce module gere le cycle complet de la nourriture des poules : achat fournisseur, entree en stock,
consommation quotidienne par lot. Les achats d'aliments utilisent les tables unifiees achats et achats_lignes
(section 6.2), partagees avec tous les autres types d'achats (vaccins, medicaments, materiel).

6.2 Tables achats et achats_lignes (communes a tous les achats)
Un seul couple de tables gere toutes les factures fournisseurs, qu'il s'agisse d'aliments, de vaccins, de
medicaments ou de materiel. La distinction se fait via produits.type sur chaque ligne. Ce design evite la
duplication et centralise toute la logique d'achat en un seul endroit.
